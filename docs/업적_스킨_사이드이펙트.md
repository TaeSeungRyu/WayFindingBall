# 업적(A) + 공 스킨(B) 콤보 — 사전 영향 분석

본 문서는 두 기능을 추가하기 **전에** 변경/위험 지점을 점검하는 사전 작업서입니다.
실제 코드는 합의 후 단계별로 진행합니다.

---

## 1. 추가/변경되는 시스템

| 항목 | 신규/변경 | 비고 |
|---|---|---|
| 배지(업적) 정의 | **신규** | 코드 상수 또는 JSON. 배지 N개, 조건/이름/설명/아이콘 |
| 잠금 해제 상태 저장 | **신규** | SharedPreferences (key 예: `unlocked_badges`) |
| 배지 평가 로직 | **신규** | 게임 종료 시 + 화면 진입 시 records 스캔 → 새로 해제된 배지 산출 |
| 공 스킨 정의 | **신규** | 코드 상수 (id, 이름, 색/추가 그림, 잠금 조건) |
| 잠금 해제·현재 선택 스킨 | **신규** | SharedPreferences (key 예: `unlocked_skins`, `current_skin`) |
| 메인 메뉴 진입점 1개 | **변경** | "내 도감" 또는 "보물상자" 추가 (배지/스킨 탭) |
| 배지+스킨 화면 | **신규** | 2-탭 화면 |
| 게임 종료 알림 | **변경** | `ResultScreen`에 "새 배지 획득!" 토스트/배너 |
| 공 렌더링 | **변경** | 현재 `BallRed` 단일 색 → 선택된 스킨에 따라 색/장식 변경 |

## 2. 기존 코드의 영향 위치

### 2-1. 공(Ball) 렌더링 영향
공 색이 곳곳에 박혀 있어 가장 많은 변경이 필요한 영역입니다.

| 파일 | 사용처 |
|---|---|
| `ui/MazeCanvas.kt` (또는 GameScreen 내 Canvas) | 게임 중 공 본체 그림 |
| `ui/GameScreen.kt` | 폭사 효과, 골인 폭죽 등에서 공 색 참조 |
| `ui/MazeEditorScreen.kt` 미리보기 | 에디터 미리보기에서도 공 렌더링 |
| `ui/theme/Color.kt` | `BallRed` 상수 정의 |

**대응**: `BallSkin` 데이터 클래스를 도입하고, 위 모든 렌더 진입점이 `currentSkin`을 받도록 통합.
공 색 외에 **추가 효과**(꼬리, 후광, 패턴)를 스킨이 제공한다면 Canvas 그리기 로직 자체 확장 필요.

### 2-2. 기록(GameRecord)에 의존하는 배지 평가
기존 `RecordsRepository`를 **읽기 전용으로** 사용하므로 안전합니다.

| 파일 | 영향 |
|---|---|
| `data/RecordsRepository.kt` | 읽기만. 변경 없음 |
| `data/GameRecord.kt` | 변경 없음 |
| 새 `data/AchievementsRepository.kt` (신규) | 잠금 해제 상태 저장/로드 |

### 2-3. 게임 종료 흐름
| 파일 | 영향 |
|---|---|
| `ui/ResultScreen.kt` | 기록 저장 직후 배지 평가 → 신규 해제 시 토스트 표시. 기록 저장 자체 로직은 그대로 둠 |
| `ui/GameScreen.kt` | 변경 없음 (onFinished 콜백 시그니처 유지) |

### 2-4. 메인 메뉴
| 파일 | 영향 |
|---|---|
| `MainActivity.kt` 네비게이션 | 새 경로(예: `Routes.COLLECTION`) 추가, 메인 화면에 진입 버튼 |
| 메인 화면 컴포넌트 (예: `ui/HomeScreen.kt`) | 버튼 1개 추가 |

## 3. 위험도 평가

| 위험 | 가능성 | 영향 | 완화 |
|---|---|---|---|
| **공 렌더링 누락 위치** (BallRed 사용처 일부만 수정) | 중 | 일부 화면만 색이 바뀌어 일관성 깨짐 | grep으로 `BallRed` 전수 파악 후 한 번에 교체. 폭죽/잔상 색은 의도적으로 빨간색 유지 가능 |
| **배지 평가 성능** (records 매번 전수 스캔) | 낮 | 기록 수백 개에서도 100ms 이내 | 화면 진입 시 1회 평가만. 이벤트 후 캐싱하면 추가 안심 |
| **신규 사용자에게 배지 표시 안 됨** | 낮 | 신규 진입 시 잠금만 보임 (정상 동작) | 명확한 잠금 조건 표시(예: "별 10개 모으면 해제") |
| **기존 사용자 데이터 마이그레이션** | 낮 | 기존 기록은 그대로 활용. 다만 첫 진입 시 기존 기록 기준으로 배지 일괄 해제 | 첫 진입 시 평가 1회 수행 + 토스트 묶음 표시 |
| **스킨 정의 변경 시 호환성** | 중 | 스킨 ID를 enum/문자열로 저장했는데 추후 제거 시 `current_skin`이 무효 ID 가리킬 수 있음 | 로드 시 미존재 ID면 기본 스킨으로 fallback |
| **순환 의존**(스킨이 배지에 의존) | 낮 | 데이터 모델 정리만 잘 하면 OK | "배지 → 스킨 잠금 해제 키" 단방향만 허용 |
| **테스트 부재** | 중 | 배지 조건 오판 가능 | 배지 평가 함수만이라도 별도 함수로 분리 → 추후 단위 테스트 용이 |

**종합 위험도**: **중간 이하**. 게임 루프나 충돌 처리 같은 민감한 부분에 손대지 않습니다.

## 4. 데이터 모델 초안

```kotlin
// 배지
data class Badge(
    val id: String,           // "first_clear", "stars_30", "infinite_10" ...
    val title: String,        // "첫 발자국"
    val description: String,  // "스테이지를 한 번 클리어해요"
    val unlockCheck: (List<GameRecord>) -> Boolean
)

// 스킨
data class BallSkin(
    val id: String,           // "default", "rainbow", "shadow" ...
    val name: String,         // "무지개 공"
    val unlockBadgeId: String?, // null = 기본 (항상 해제). 아니면 그 배지 획득 시 해제
    // 렌더링 정보:
    val coreColor: Color,
    val accentColor: Color,
    val decoration: BallDecoration  // NONE, GLOW, TAIL, PATTERN_STAR, ...
)
```

저장 형식 (SharedPreferences):
```
key: "unlocked_badges" → JSON ["first_clear", "stars_30"]
key: "unlocked_skins"  → JSON ["default", "rainbow"]
key: "current_skin"    → "rainbow"
```

## 5. 권장 진행 순서

1. **배지 정의 + Repository** (코드만, UI 없음)
2. **배지 평가 함수 + ResultScreen 토스트** (간단한 모달 또는 한 줄 토스트)
3. **"내 도감" 화면 — 배지 탭만** 먼저
4. **공 스킨 데이터 모델 + Repository**
5. **GameScreen 공 렌더링 통합** (가장 큰 변경. 단계 4와 분리해서 진행)
6. **"내 도감" — 스킨 탭 추가 + 선택 UI**
7. 메인 메뉴에 진입점 추가 + QA

각 단계는 독립 커밋 단위로 가능하며, 중단해도 앞 단계만으로 기능적입니다.

## 6. 결정이 필요한 항목

다음 안에 동의/수정 필요:

- [ ] 배지 개수: **8~12개** 권장 (너무 많으면 의미 흐려짐)
- [ ] 스킨 개수: **6~10개** 권장
- [ ] 메인 메뉴 진입점 이름: "내 도감" / "보물상자" / "내 컬렉션" 중
- [ ] 스킨 디자인 방향: 색만 바꾸는 가벼운 안 vs 꼬리/후광 같은 효과 포함 안
- [ ] 토스트 위치/형태: ResultScreen 상단 배너 vs 별도 모달

위 항목들에 답을 주시면, 바로 1단계부터 시작할 수 있습니다.
